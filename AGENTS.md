## Learned User Preferences

- For UI, **default to shadcn/ui**: look under `components/ui/` for the primitive; if it is not installed yet, add it with `bunx shadcn@latest add <component>` (then compose from it) instead of hand-rolling tables, dialogs, forms, etc., unless the user explicitly asks for a custom implementation. Prefer shadcn **Form** patterns (react-hook-form + primitives) over ad-hoc deprecated `FormEvent` flows unless there is a specific reason not to.
- When using reference screenshots for UI, treat them as layout/hierarchy guidance unless the user asks for a pixel-perfect copy; preserve the app's existing color themes and typography when the user clarifies that the reference is not literal.
- For catalyst research, avoid hardcoded ticker/company-specific event maps, deterministic bucket/checklist search plans, and ticker-specific discovery vocabulary; use open-ended hosted search breadth and general discovery patterns that surface company-specific strategic milestones from current news—not only official calendar events—for any proper company.
- When debugging missing catalyst results or empty tool output or “silent” model behavior, do not attribute outcomes to cache without evidence; prioritize retrieval, extraction, and provider/AI-SDK paths first; use Convex function logs or a local script that mirrors the same `generateText` call before speculative fixes — Convex **actions** do not run on the developer’s machine, so `localhost` ingest or similar will not see those requests.
- For **GPT-hosted web research**, keep **GPT as the orchestrating model** and use **Exa’s `webSearch` tool** (`@exalabs/ai-sdk`) with the Vercel AI SDK (`generateText`, `tools.webSearch`, `stopWhen`) instead of OpenAI’s built-in web search unless native OpenAI search is explicitly required; set **`EXA_API_KEY`** where Exa runs (e.g. Convex actions).
- For catalyst events, keep **`summary`** and **`whyItMatters`** concise in extraction prompts and structured-output guidance (roughly one to two short sentences for `summary`, one short sentence for `whyItMatters` unless strictly necessary).
- In catalyst results UI, prefer a concise **table** (`Table` / related primitives) instead of sprawling card-heavy layouts unless a screen truly needs cards; show **one clickable link per source** when an event has multiple sources rather than collapsing into a single “N links” control; keep **research loading / blocker states on the ticker route** visually aligned with the home hero (narrow `max-w-5xl` column, viewport-centered shell via `lib/research-route-layout.ts`).
- Saving catalyst research from results to the portfolio should be **all-or-nothing** for that run — do **not** ask users to pick individual catalyst rows.
- On first load or full refresh of the home page, focus the ticker search **`Input`** (e.g. `autoFocus` **after** `{...field}` from react-hook-form so the spread does not override it).
- In catalyst **extraction**, when sources describe the **same dated or named real-world occasion** (e.g. a flagship conference and expected announcements there), output **one merged event**; put product or launch detail in `summary` / `whyItMatters` and avoid separate rows that only differ by angle or `eventType`.
- For catalyst **extraction**, include the **canonical issuer name** (e.g. from Finnhub) in the prompt when available—especially for **short or ambiguous tickers**—so lock-up, float, and insider-supply rows are not conflated with the wrong company’s facts.
- For **lock-up, float, insider-supply, and similar corporate milestones**, do **not** merge **large share counts**, **percentages**, and **calendar anchors** from **different** snippets into one event unless **one** cited source states them together; conflicting dates across sources should yield **separate events** or a single lower-confidence row, not a stitched hybrid.

## Learned Workspace Facts
- Use **Bun** for installs and scripts (`bun install`, `bun run …`); `bun.lock` is the only committed lockfile—do not add `package-lock.json`.
- Stonkseer: users research upcoming catalysts (next 12 months) for a ticker and can save to a portfolio. The app uses Next.js and React with shadcn UI, Convex for backend and reactivity, Clerk Google SSO, and the Vercel AI SDK for structured research. Starting research from **`/`** navigates to **`/${normalizedTicker}`** (`app/(main)/[ticker]/`); **`sessionStorage`** key **`stonkseer:activeResearch:{SYMBOL}`** bridges **`{ runId, anonymousTokenHash? }`** so the ticker page can subscribe to Convex while anonymous auth stays off the URL.
- Global client providers (Convex+Clerk bridge, theme, auth sync, sidebar tooltip provider, and similar) live under `components/providers/`; wire the tree in `components/providers/app-providers.tsx` and keep `app/layout.tsx` thin. The main app shell/sidebar live in `components/main-shell.tsx` and `components/app-sidebar.tsx`, with route pages under `app/(main)/`.
- The MVP intentionally allows one anonymous ticker research trial before sign-in; saving to a portfolio requires Google sign-in. User-initiated research reuses a prior completed run for the same ticker for **7 days** when strategy version and event data match (`RESEARCH_CACHE_TTL_MS` in `convex/research.ts`); cache hits do not count toward quotas. Signed-in users are capped at **10** non-cache research runs per rolling **24 hours** and at most **2** concurrent in-progress runs (`convex/research.ts`).
- Anonymous research protection uses layered controls: deterministic ticker normalization/validation, hashed IP bucket, signed browser token, cache reuse, and invalid tickers must not consume the anonymous trial; anonymous runs use the **same hosted-search query breadth** as signed-in runs (no separate anonymous web-query budget), while **daily anonymous non-cache quotas** still apply (per IP and signed-browser-token gates in `convex/research.ts`).
- The anonymous research endpoint is a Next.js Route Handler because it needs request headers, IP-derived signals, and HTTP-only cookie handling; authenticated portfolio saves should use Convex mutations.
- Research runs in Convex actions: validate tickers deterministically using Finnhub market data when `FINNHUB_API_KEY` is set; **`warnExpectedEnvMissing`** logs **`console.warn` once per warm isolate** when expected research env vars are absent (`FINNHUB_API_KEY`, `AI_GATEWAY_API_KEY`/`VERCEL_OIDC_TOKEN`, `EXA_API_KEY`, enabled provider model ids)—not invalid-key checks. Finnhub profile/industry is **company context** in the merge prompt, while Finnhub profile/earnings/**earnings-calendar** snippets remain in the **evidence pool** (`finnhub_metadata` provenance). **All** enabled hosted search providers in `CATALYST_HOSTED_SEARCH_PROVIDERS` run **in parallel** each run (**Gemini** `google.tools.googleSearch`; **OpenAI** GPT orchestrator + Exa `webSearch`; **Anthropic** `webSearch_20250305`; **xAI** `x_search`). Each provider produces a **full catalyst research report** (`buildCatalystReportPrompt`—chat-quality synthesis with timing, status, why-it-matters, and source URLs; no bucket checklists or company-specific vocabulary) plus **tool-excerpt-only snippets** and a `seenUrls` list; the old digest round-robin quote fallback (`digest_fallback`) is **deleted**—never attribute model digest text to URLs. After round 1, a **dynamic follow-up pass** ([`lib/research-followup.ts`](lib/research-followup.ts)) has a model (env `CATALYST_FOLLOWUP_MODEL`, falls back to `AI_GATEWAY_MODEL`) write up to **6** targeted queries (`CATALYST_FOLLOWUP_MAX_QUERIES`) from the merged reports—never hardcoded—run via the **Exa search API** (`EXA_API_KEY`, provenance `follow_up_search`). Web evidence caps: **64** snippets (`MAX_WEB_SNIPPETS`) at **2500** chars per quote (`WEB_SNIPPET_QUOTE_LENGTH`, including Exa `maxCharacters`), after URL dedupe and domain diversification. Optional **Exa Contents** deep-read on up to **8** URLs ranked by provider hits + excerpt length ([`lib/research-exa.ts`](lib/research-exa.ts)). The **merge/extraction** step via Vercel AI Gateway (`AI_GATEWAY_MODEL`) receives the **provider reports plus evidence snippets** (keyword `candidates` hints are removed) and uses **`formatResearchBreadthExtractionBlock`** for cross-family coverage. Then **provenance-aware grounding** ([`lib/research-grounding.ts`](lib/research-grounding.ts)): snap source quotes to snippet excerpts by URL (`evidence_snippet`); keep sources whose URL any provider saw but lacks an excerpt as **`report_derived`** (model quote kept); drop only sources/events citing URLs nobody saw. Finnhub earnings-calendar rows merge in as baseline earnings events when extraction omits them. Default hosted search steps/uses: **8** (`CATALYST_OPENAI_EXA_AGENT_MAX_STEPS`, `CATALYST_ANTHROPIC_WEB_SEARCH_MAX_USES`). Strategy version [`RESEARCH_STRATEGY_VERSION`](lib/research-strategy.ts) busts cache on pipeline changes (now `report-merge-v8`). For Anthropic hosted search, use `anthropic.tools.webSearch_20250305`; `webSearch_20260209` produced an unsupported `anthropic-beta` header in this Gateway path.
- The GitHub repository is `https://github.com/0xGuvnor/stonkseer`, with `master` tracking `origin/master`; the Vercel project is `0xguvnors-projects/stonkseer` at `https://stonkseer.vercel.app`.
- The Convex dashboard project is named `stonkseer`, and the local CLI has been connected to it; the Clerk JWT issuer for Convex belongs in Convex deployment environment variables, while Clerk frontend/server keys belong in local Next.js/Vercel environment variables; if real Clerk keys are set, keyless `.clerk/` fallback is not required and generated `.clerk/` content stays out of git.
- On Vercel, `NEXT_PUBLIC_*` values (including `NEXT_PUBLIC_CONVEX_URL`) are inlined at build time, so they must be set in the project's environment variables for each environment that production or preview builds use.
- Convex **actions** execute on Convex’s servers, not on a developer’s laptop: logging or `fetch` to `127.0.0.1` from inside an action will not hit the user’s local machine; use the Convex dashboard / function logs or a local repro for evidence.
- xAI **Grok X** (`x_search`) via **Vercel AI Gateway** (`CATALYST_XAI_SEARCH_MODEL` as a gateway model id): the SDK may return **`tool-call`** and **`source`** URL parts without normalized **`tool-result`** rows, so `result.toolResults` can be empty even when search ran; derive X/Twitter snippets from parsed **`x_search` post text** only—do **not** attribute the model digest to X URLs. With **Vercel AI SDK 6**, top-level **`result.toolResults` reflects the last step only**—merge across **`result.steps`** when collecting posts, and if **no** `x_search` tool results appear in steps, **also** scan top-level **`result.toolResults`** for Gateway edge cases. Disabling **`enableImageUnderstanding` / `enableVideoUnderstanding`** saves cost and latency but can **reduce relevance** on posts that lean on images or video; env-gate or turn media back on if X evidence quality drops. Direct `xai.responses()` is a different auth path (`XAI_API_KEY`); do not assume it when the product standard is Gateway-only.
